# OPEN BUG: data-binding web components don't register/update reliably

## ROOT CAUSE FOUND (2026-06-06, via production preview)
In the PRODUCTION build (`bun run build` + `astro preview`), `<talos-gauge>` is
**not registered at all**: `customElements.get('talos-gauge') === undefined`,
elements are `:defined === false`, no shadow DOM. The bare side-effecting import
`import "@shelfwood/talos-ui/wc"` in the homepage `<script>` is NOT executing /
its `customElements.define()` side effect is being dropped by the Astro/Vite
bundler (bare import treated as side-effect-free, or the define block tree-shaken
out of dist/wc/index.js).

This is almost certainly the SAME root cause as the dev "frozen at 40" symptom:
the module's registration/side-effects aren't running the way the code assumes.
(Earlier in the session the very first instrument-test DID animate live — so it
*can* work; something about the import/bundle path is non-deterministic.)

## UPDATE 2 — registration FIXED, reactivity STILL broken
- Added the wc entry to package.json `sideEffects` → `customElements.define`
  now survives tree-shaking. Confirmed in prod preview:
  `customElements.get('talos-gauge') === true`, elements upgrade, INITIAL render
  is correct. **Commit this fix — it is correct and necessary.**
- BUT a SECOND, independent bug remains: `attributeChangedCallback` never fires
  (accFired:0, renderCalled:0) even on a direct `setAttribute` in clean prod
  (no HMR). Confirmed:
    - acc + render ARE on the prototype (hasOwnProperty true), not the instance.
    - observedAttributes getter returns the full array on the registered ctor.
    - a trivial inline probe element's acc DOES fire in the same page.
    - ctor is minified to `T` (esbuild minify on).
  Everything inspectable looks correct, yet the browser never invokes acc for
  these elements. Likely an esbuild class-transform/minify interaction with
  custom-element lifecycle that I could not close in reasonable time.

## RECOMMENDED NEXT ACTION (reactivity)
Switch the primitives off `attributeChangedCallback` onto a **MutationObserver**
watching `attributes` (the mechanism `<talos-panel>` already uses successfully in
this exact environment for childList). In connectedCallback:
  this._mo = new MutationObserver(() => this.scheduleRender());
  this._mo.observe(this, { attributes: true });
and drive value tweens from there. Disconnect in disconnectedCallback. This
sidesteps whatever is suppressing attributeChangedCallback and is known-good
here. Apply to gauge/trend/meter/flow. (Alternatively: try tsup with
`--keep-names` / minify off to test the esbuild-transform hypothesis first — but
the MutationObserver route is the reliable fix.)

## Original symptom notes (dev mode) — kept for reference


## Symptom
On the homepage console, every data-binding primitive renders its INITIAL value
correctly but never updates as the sim loop drives it:
- `<talos-gauge>` readout + band colour frozen at the initial `value` (e.g. 40),
  even though the `value` ATTRIBUTE is observed changing live (e.g. 89).
- `<talos-trend>` readout frozen too — and it uses imperative `push()`, NOT
  attributeChangedCallback. **So the failure is NOT specific to the attribute
  path.** Both the attribute path and the imperative path are stuck.

## Established by direct DOM probing (facts, not theory)
1. Elements ARE upgraded (`:defined`, ctor = TalosGauge), shadow DOM present.
2. `observedAttributes` is correctly registered (returns the full array); I also
   converted it to a `static get` getter — the registered ctor shows
   `observedIsGetter: true` returning the right array.
3. DESPITE that, the gauge's `attributeChangedCallback` never fires on
   setAttribute (`accFired: 0`), even on a freshly `createElement`'d gauge.
4. A trivial inline probe custom element defined in the SAME page DOES fire its
   attributeChangedCallback (probeAccFired: 2). **So the environment/browser is
   fine; the issue is specific to the package elements.**
5. No JS errors in console (only Vite HMR noise — but note MANY HMR reconnects
   and a "server connection lost" from repeated dev restarts this session).
6. `render()` and `tweenTo()` were monkeypatch-counted: 0 calls over 1.5s while
   the sim ran. So render simply never re-runs after construction.

## Leading hypotheses (unverified — start here)
A. **Stale module graph / double-registration via Vite HMR.** Repeated dev-server
   restarts mid-session left the tab on an old module graph. `define()` is
   idempotent (`if (!customElements.get(name))`), so if an OLD class registered
   `talos-gauge` first, the new getter-fixed class is ignored — yet the descriptor
   showed the getter, which argues against this. NEEDS a clean browser context
   (fully quit/reopen the tab, or test the production `bun run build` + `preview`,
   NOT dev) to rule out HMR entirely. **Try this FIRST — cheapest.**
B. **The sim tick throws inside the gauge's setAttribute reaction**, aborting the
   rest of tick() before `trend.push()` — would explain both freezing together
   (gauge attr is set on line 1, trend push on line 2). But console shows no
   error. Wrap the gauge update in try/catch in the page and log, OR move
   trend.push to the TOP of tick() and see if trend then updates while gauge
   doesn't — that isolates whether it's an abort vs a render failure.
C. **render() throws on non-initial calls** (e.g. a null query, a NaN from an
   attribute) and the throw is swallowed by the rAF callback. Add a try/catch
   with console.error inside render()/scheduleRender to surface it.

## Recommended fix path
1. Rule out HMR: production build + `astro preview`, fresh tab. If it works
   there, the bug is purely dev-HMR staleness — document and move on.
2. If it persists: add try/catch + console.error around render() and the rAF
   step in talos-gauge; run the sim; read the real exception.
3. If attributeChangedCallback genuinely never fires for these elements but does
   for a probe, switch reactivity to a MutationObserver on attributes (the
   mechanism `<talos-panel>` already uses successfully for childList) — known to
   work in this environment.
